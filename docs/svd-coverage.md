## Reading the numbers — two modeling styles

The auditor credits a register when its byte offset appears as a literal (or
inside a range/array handler) in the peripheral's source. That is an exact
proxy **only** for peripherals modeled with explicit per-register `match`
arms. The RT1060 emulator uses two backing styles, and the percentage means
different things for each:

1. **Explicit per-register models** (`match offset { 0x14 => …, _ => 0 }`).
   Unhandled offsets genuinely read 0 / drop writes, so the auditor's count is
   the true coverage. All GPIO / LPUART / ADC / GPT / PIT / SRC / DCDC / WDOG
   and the LPI2C / LPSPI buses are modeled this way.

2. **Whole-window backing** — a flat offset-indexed array (`regs[off>>2]`,
   `mem[off]`) or the CCM_ANALOG SET/CLR/TOG alias quad. **Every** register in
   the block reads back what firmware wrote by construction, so per-register
   coverage is 100% functionally even though the auditor, seeing no per-offset
   literal, undercounts. CCM, CCM_ANALOG, SEMC, FLEXSPI, IOMUXC(_GPR), and the
   FlexPWM / QTMR blocks are backed this way. Their reported < 100% is an
   audit artifact, not a missing register (the status bits firmware actually
   polls — CCM `CDHIPR`, SEMC/FLEXSPI `STS0` idle, PLL `LOCK` — are forced in
   explicit arms and verified separately).

## Genuine in-scope surface gaps (8) — the v2 checklist

These use explicit per-register models, so the unhandled offsets truly read 0.
Each is either an unmodeled *mode* or an *instant-transfer* timing knob with no
observable effect at this emulator's transaction fidelity.

- **LPI2C1..4** (14/28 each) — the **slave register block** (`SCR`, `SSR`,
  `SIER`, `SDER`, `SCFGR1/2`, `SAMR`, `SASR`, `STAR`, `STDR`, `SRDR`) is
  unmodeled: the SwiftIO / Zephyr / Arduino targets drive LPI2C as **I²C
  master only** (behavioural — START/addr/data/STOP + host-answered reads).
  Master timing/config (`MCFGR2` glitch filter, `MDMR` data-match, `MCCR0`
  clock high/low) is *register-surface*: at instant transfer it has no timing
  effect (a v2 item is to store/read-back these instead of dropping them).
- **LPSPI1..4** (15/17 each) — `DMR1` (data-match 1) and `CCR` (clock config:
  `SCKDIV`/`DBT`/`PCSSCK`/`SCKPCS` delays) are *register-surface*: data-match
  triggering is unmodeled and the shift is instantaneous, so the clock-delay
  fields have no observable effect. Store/read-back is the v2 item.

## Out-of-scope stubs (67)

Deliberately deferred per the build contract — routed to `RawRegs` readback or
the warn-once `unknown` window as the boot path touches them. Highlights: the
full **DMA0** (528 regs) / **DMAMUX** engine is *behaviourally* modeled (TCDs
move data through the bus) but its register map is not per-register audited;
likewise **USB1/2**, **FlexCAN1..3**, **SAI1..3**, **USDHC1/2** boot their real
drivers without a full SVD audit. **ENET/ENET2**, **USBPHY**, **ADC_ETC**,
**XBAR**, **LCDIF/PXP/CSI**, **DCP/TRNG/BEE** crypto, **GPIO10**, and the
`PMU`/`TEMPMON`/`XTALOSC24M` analog aliases are unmodeled. See
[`ROADMAP.md`](../ROADMAP.md).
